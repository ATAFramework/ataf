ATAF

Agent Tool Acquisition Framework — agents that acquire new tools at runtime.

Design v1 · locked 2026-06-26 Apache-2.0 v0.1 in progress Python · FastAPI · SQLite

1Concept

Every agent framework today — LangChain, CrewAI, the Claude Agent SDK, MCP — assumes the agent's toolset is hand-curated at build time. If the agent needs a capability the developer didn't anticipate, the agent fails.

The insight

The LLM itself is the best discovery + synthesis engine. Instead of hunting external registries for a missing capability, just ask the LLM to write the missing tool, then deploy that tool live so the agent can call it on the very next turn.

The agent fetches its current tool catalog from an ATAF server and asks an LLM how to perform a task. The LLM either picks an existing tool, writes a new Python tool inside <ATAF param="newtool">…</ATAF> tags, or declines. On a proposal, the agent forwards the code to the ATAF server, which materializes it into a live FastAPI endpoint and returns the new tool's schema. The agent re-prompts the LLM with the refreshed catalog, and the LLM calls the new tool. Tools persist indefinitely and are reusable across every agent talking to that server.

🧩

Acquire, don't curate

Tools materialize on demand from LLM-authored code — no pre-registration.

♻️

Persist & share

A built tool survives restarts and is reusable by all agents on the server.

🛡️

Human-gated

New tools start PENDING_REVIEW; a human approves before invocation.

4The LLM Contract — Three Responses

The agent's prompt carries the user task, the current catalog (with status flags), and the tag protocol. The LLM must answer in exactly one of three shapes.

①

Use existing tool

A native tool-call in the LLM's own format (Claude tool_use, OpenAI function_call).

②

Propose new tool

A function inside <ATAF> tags — typed params, return annotation, docstring, stdlib only.

③

Decline

Cannot do the task and cannot write a tool for it. The agent halts the loop.

Response ② — the proposal shape

<ATAF param="newtool">
def circle_area(radius: float) -> float:
    """Compute the area of a circle.

    Args:
        radius: The radius of the circle, in any unit.

    Returns:
        The area in the squared unit of the radius.
    """
    import math
    return math.pi * radius ** 2
</ATAF>
Hard requirements (v0.1)

Exactly one top-level function · every parameter type-annotated · a return annotation · a Google/NumPy-style docstring · imports from the Python standard library only. The builder rejects anything else with a 400 and a stable error code.

6Concurrency — One Global Build Lock

Only one tool builds at a time, server-wide. Invocation of existing tools is never blocked.

stateDiagram-v2
    direction LR
    [*] --> Idle
    Idle --> Building: propose acquires lock
    Building --> Idle: deploy done · notify waiters

    state Building {
      direction LR
      validate --> schema
      schema --> persist
      persist --> register_route
    }
A global lock sidesteps "are these the same tool?" — no name normalization, no semantic similarity.
While a build holds the lock

POST /tools/propose → 202 WAIT  ·  GET /tools → long-poll (blocks until done)  ·  POST /tools/{id}/invoke → proceeds normally (execution is never blocked).

Locked · global, not per-tool

Per-tool locks need to identify "the same tool" across submissions — fragile name normalization or unreliable similarity. Builds are infrequent and sub-second, so a global lock is an acceptable v0.1 trade.

Locked · no semantic dedup

If two agents propose circle_area and area_of_circle, both deploy. Duplicates accumulate; humans prune them via approve/reject. No false-positive merges.

Long-poll mechanics

On 202 WAIT the agent polls poll_url. The server holds the connection until the build completes (returns the refreshed catalog) or a 30s timeout (408 → retry). On receiving the refreshed catalog, the agent must re-prompt the LLM before doing anything else — this refresh is hidden inside the client library, so user code simply sees "the tool list now includes the new tool."

7Tool Lifecycle

The registry is append-only with status mutations — tools are never auto-deleted, which keeps the system inspectable and undoable.

stateDiagram-v2
    direction LR
    [*] --> Submitted: agent POSTs code
    Submitted --> Rejected400: validation fails
    Rejected400 --> [*]
    Submitted --> PendingReview: built & deployed
    PendingReview --> Authorized: admin approve
    PendingReview --> Unauthorized: admin reject
    Authorized --> Invokable: forever
    Unauthorized --> Kept: in registry, never invokable
    Invokable --> [*]
    Kept --> [*]
Validation rejection is the only path that doesn't persist a row. Everything past deploy lives forever.

On-disk layout

ataf_data/
  ataf.sqlite                  # metadata index (one row per tool)
  tools/
    circle_area_v1.py          # the actual generated code — git-trackable
    rectangle_area_v1.py
  logs/
    deployment.jsonl           # append-only audit: propose · deploy · approve · …

On startup the registry reloads every row, re-reads its .py file, and verifies the stored code_sha256. A missing file or hash mismatch flips the tool to UNAUTHORIZED with a warning — never an auto-delete.

8Governance — Two-Layer Enforcement

StatusSet byLLM sees it asServer allows invocation?
PENDING_REVIEWDefault after deploy"Unavailable — pending review"Config flag allow_pending_invocation (default false)
AUTHORIZEDHuman adminNormal toolYes
UNAUTHORIZEDHuman admin"Unavailable — rejected"No

Layer 1 · Prompt-level

The catalog sent to the LLM carries each status flag. The prompt instructs: non-AUTHORIZED tools are unavailable — do not propose a duplicate; decline instead. This prevents the duplication loop.

Layer 2 · Server-level

Even if the LLM hallucinates and invokes a non-authorized tool, the server returns 403 TOOL_NOT_AUTHORIZED. The agent forwards it; the LLM is instructed to stop calling tools and return its final answer.

Why both

Layer 1 is the happy path that keeps the LLM cooperative; Layer 2 is the hard safety net. v0.1 trusts the LLM is honest — code sandboxing is deferred to v0.3.

9Build Pipeline — Inside builder.py

The builder is a pure, lock-free unit: it validates, generates the schema, and persists. Lock acquisition and live-route registration stay in main.py, so the builder is fully unit-testable without a running server.

Two failure paths: validation rejects bad code (400); a server-side fault after validation is retryable (500 · TOOL_NOT_DEPLOYED).
Two kinds of failure

Validation failures are the agent's fault — malformed code — and return 400 with a specific code (below). A deployment failure means valid code that the server couldn't persist (disk full, locked DB); it returns 500 TOOL_NOT_DEPLOYED and the same code may succeed on retry. Both log a deploy_failed audit event.

Schema generation

Hand-rolled from ast + a small Google-style docstring parser — no griffe / docstring-parser dependency in v0.1. Maps Python annotations to JSON-schema types (int→integer, float→number, list[T]→array, dict→object), unwraps Optional[T] / T | None, and marks defaulted parameters optional. Unknown annotations degrade to an unconstrained schema rather than rejecting a valid tool.
